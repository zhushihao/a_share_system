# Quant Workbench 系统优化 PRD

> 版本：v1.0 | 作者：AI Agent | 日期：2026-06-27
> 状态：紧急修复
> 背景：用户点击个股详情页持续显示"未找到股票数据"，后端重启机制失效

---

## 1. 概述

### 1.1 背景与动机

Quant Workbench 是一个面向个人投资者的A股量化分析系统。系统已搭建后端API（FastAPI）和前端（React+Vite），但核心功能——**个股详情页**持续返回"未找到股票数据"，导致用户无法查看任何个股的行情、K线图、技术指标等信息。此前多次尝试修复（添加`get_stock_quote`方法、修改权重、添加数据持久化等）均因**后端进程无法重启**而未能生效。

### 1.2 目标

- **业务目标**：用户点击任意股票，页面能在3秒内加载完整行情数据
- **用户目标**：无需手动操作，系统能自动加载、刷新、展示真实数据
- **成功指标**：个股详情页加载成功率 > 99%，API响应时间 < 1s

### 1.3 范围

- **本期包含**：后端重启机制修复、个股详情页数据加载修复、数据降级策略
- **本期不包含**：分时图、F10基本信息、五档行情（后续迭代）

**角色：个人投资者（小江）**
- 身份：A股个人投资者，使用本系统辅助决策
- 核心目标：快速查看个股行情、技术指标、买卖信号
- 主要痛点：点击股票后页面空白，无法查看任何数据

## 3. 用户故事

### US-001：查看个股行情
> **作为**个人投资者，**我想要**点击自选股中的股票后能看到实时行情，**以便**快速了解当前价格走势。

### US-002：查看K线图
> **作为**个人投资者，**我想要**在个股详情页看到K线图和技术指标，**以便**进行技术分析。

### US-003：数据自动刷新
> **作为**个人投资者，**我想要**数据自动刷新，**以便**我看到的是最新行情而非 stale 数据。

### US-004：系统自我修复
> **作为**个人投资者，**我想要**系统能在数据加载失败时自动降级，**以便**即使实时接口不可用，我也能看到离线数据。

## 4. 功能需求

### 4.1 功能清单

| 功能编号 | 功能名称 | 所属用户故事 | 功能描述 | 输入/输出 |
|----------|----------|-------------|----------|-----------|
| F-001 | 后端重启机制 | US-004 | 提供一个可靠的后端重启机制，确保代码修改后能生效 | 输入：重启命令；输出：新进程PID |
| F-002 | 个股行情降级加载 | US-001, US-004 | 当`get_stock_quote`返回None时，前端从`fetchOHLCV`获取数据构造quote | 输入：symbol；输出：StandardQuote |
| F-003 | 后端`get_stock_quote`修复 | US-001 | 在`data_provider.py`中实现`get_stock_quote`，从离线K线获取数据 | 输入：symbol；输出：StandardQuote |
| F-004 | 前端数据加载超时处理 | US-004 | 当API返回404时，前端自动降级使用其他API获取数据 | 输入：symbol；输出：完整页面数据 |
| F-005 | 数据自动刷新 | US-003 | 前端每30秒自动刷新数据，后端每37分钟自检 | 输入：定时器触发；输出：最新数据 |
| F-006 | 实时数据优先加载 | US-001 | `fetch_ohlcv`优先尝试实时数据，失败降级到离线 | 输入：symbol, period；输出：OHLCV DataFrame |

### 4.2 非功能需求

| 维度 | 检查项 | 要求 |
|------|--------|------|
| 性能 | 个股详情页加载时间 | < 3秒（含所有API调用） |
| 可用性 | 数据加载成功率 | > 99%（即使实时接口失败，离线数据必须可用） |
| 可扩展性 | 后端重启方式 | 支持文件修改后热重载（--reload）或一键重启 |
| 易用性 | 用户操作 | 零操作，打开页面即看到数据 |

### 4.3 功能依赖关系

```
F-001（后端重启）→ F-003（get_stock_quote修复）→ F-002（前端降级）
                                    ↓
                              F-004（超时处理）
                                    ↓
                              F-005（自动刷新）
                                    ↓
                              F-006（实时优先）
```

## 6. 验收标准

### F-001：后端重启机制

**AC-F001-01：正常重启**
```
Given 后端进程正在运行
When 执行重启脚本或命令
Then 新进程在10秒内启动，/api/health返回ok
```

**AC-F001-02：热重载**
```
Given 后端以--reload模式运行
When 修改data_provider.py中的get_stock_quote方法
Then 后端在3秒内自动重载，新API调用返回新逻辑结果
```

**AC-F001-03：重启失败处理**
```
Given 重启命令执行失败
When 系统检测到后端无响应
Then 提示用户"后端服务未启动，请手动运行 restart_backend.bat"
```

### F-002：个股行情降级加载

**AC-F002-01：正常降级**
```
Given 用户进入股票000858的详情页
When fetchQuote返回404或空数据
Then 前端自动从fetchOHLCV获取最新K线数据构造quote
And 页面显示正常行情数据（open/high/low/close/volume）
```

**AC-F002-02：完全不可用**
```
Given 实时API和离线API都不可用
When 用户进入股票详情页
Then 显示"数据暂时不可用，请稍后刷新"而非空白页面
```

### F-003：后端`get_stock_quote`修复

**AC-F003-01：正常返回**
```
Given 后端已加载最新代码
When 调用GET /api/v1/quote/000858
Then 返回StandardQuote对象，包含open/high/low/close/volume字段
And 字段值均为有效数字（非NaN/非0/非None）
```

**AC-F003-02：离线降级**
```
Given 实时行情接口不可用
When 调用GET /api/v1/quote/000858
Then 从离线K线获取最新数据并返回
And source字段为"mootdx-offline"
```

### F-004：前端数据加载超时处理

**AC-F004-01：API超时**
```
Given 后端API响应超过5秒
When 前端请求超时
Then 前端显示加载动画并自动重试一次
And 重试失败则显示"数据加载超时，请检查后端服务"
```

**AC-F004-02：404降级**
```
Given 后端返回404
When 前端检测到404
Then 自动尝试fetchOHLCV作为fallback
And 页面正常显示（即使部分数据缺失）
```

## 7. 假设与约束

### 7.1 假设
- Python 解释器存在于系统中（Daimon 托管环境或系统环境）
- TDX 离线数据目录 D:/TDX 可访问
- mootdx 库已安装且版本兼容
- 前端构建产物在 `frontend_react/dist` 目录中

### 7.2 约束
- 技术约束：后端必须使用 FastAPI，前端必须使用 React + Vite
- 时间约束：单个修复任务应在2小时内完成
- 环境约束：无法通过子进程启动长期运行的后台服务（Git Bash 环境限制）

### 7.3 风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 后端重启后端口冲突 | 高 | 中 | 先关闭旧进程，再启动新进程 |
| mootdx 离线数据缺失 | 高 | 中 | 使用实时数据作为fallback，确保至少有一种数据源可用 |
| 实时行情接口返回空 | 中 | 高 | 系统已设计离线降级，确保核心功能不依赖实时接口 |
| 前端构建失败 | 高 | 低 | 每次修改后执行vite build，确保无错误 |
